# Короткий звіт про Retrieval Augmented Generation (RAG)

## 1. Що таке RAG

Retrieval-Augmented Generation (RAG) — це підхід, у якому **велика мовна модель (LLM) під час відповіді додатково звертається до зовнішніх джерел даних** (документів, баз знань, вебу). Спершу спеціальний модуль-пошуковик (retriever) знаходить релевантні фрагменти тексту, потім LLM генерує відповідь, спираючись на ці фрагменти. 

Ідея вперше системно сформульована в роботі Lewis et al. (NeurIPS 2020) як поєднання **параметричної пам’яті** моделі й **непараметричної пам’яті** у вигляді зовнішнього корпусу документів. Вендорні гайди Microsoft Azure, AWS, Google Cloud у 2024–2025 роках дають майже ідентичне визначення: RAG — це спосіб «заземлити» відповіді LLM на актуальних і приватних даних через retrieval.

---

## 2. Основні архітектурні патерни

### 2.1. Classic RAG

- **Схема:** `запит → retriever → top‑K документів → конкатенація в prompt → LLM → відповідь`.
- Базовий варіант: DPR (dense retriever) + генератор (BART/LLM). 
- Підходить для: FAQ‑ботів, пошуку по документації, базового enterprise QA.

### 2.2. Покращені retrievers

- **Dense retrievers:** ембеддинги (DPR, E5, OpenAI embeddings тощо) замість чистого BM25.
- **Hybrid retrievers:** поєднання BM25 + dense для кращого покриття.
- **ColBERT / late interaction:** токен‑рівнева семантика, висока точність для складних запитів.
- **Rerankers:** cross‑encoders або LLM‑rerankers, що переупорядковують top‑N документів перед подачею в LLM, зменшуючи шум у контексті.

### 2.3. Multi‑hop / iterative RAG

- Для складних запитів, які вимагають кількох кроків міркування.
- LLM генерує проміжні підзапити, виконує декілька раундів retrieval, потім агрегує знайдені факти.
- Типові датасети: HotpotQA, інші multi‑hop QA.

### 2.4. Agentic / tool‑augmented RAG

- LLM виконує роль **агента**, який сам вирішує, які інструменти викликати: retriever, веб‑пошук, SQL‑бази, інші API.
- Поширені патерни: ReAct, plan‑and‑execute, self‑grading агенти, multi‑agent RAG (planner, retriever‑agent, validator).
- Дає кращу адаптивність: агент може переформульовувати запит, повторно робити retrieval, перевіряти власні відповіді.

### 2.5. Memory / graph‑based RAG

- **Memory‑augmented RAG:** окремий індекс довгострокової пам’яті (історія діалогу, профіль користувача), по якому теж робиться retrieval.
- **Graph RAG:** замість «плоского» тексту знання представлені як граф (ентіті, зв’язки). Пошук включає:
  1. витяг ентіті з запиту;
  2. пошук релевантних вузлів/шляхів у графі;
  3. побудову текстового summary з сабграфа;
  4. генерацію відповіді на основі цього summary.
- Особливо корисно в доменах із насиченими зв’язками (медицина, право, рекомендаційні системи).

## 3. Типові задачі та use‑cases

1. **Enterprise QA та пошук по документації.**  
   - Внутрішні чатботи по Confluence/SharePoint, політиках компанії.  
   - Практичні приклади: Thomson Reuters (RAG‑асистент для операторів підтримки), Pinterest (внутрішній асистент для SQL/аналітики).

2. **Бізнес‑аналітика.**  
   - RAG‑LLM читає новини, фінансові звіти, PDF‑звіти й відповідає на запитання бізнес‑користувачів.

3. **Технічна документація й code‑assistant.**  
   - Пошук по API‑доках, внутрішньому коді; генерація прикладів, пояснення legacy‑коду. 

4. **Спеціалізовані домени.**  
   - Медицина: Medical Graph RAG, IBM Watson Health‑подібні системи для клінічних рішень.  
   - Право: legal research assistants на основі RAG, LegalBench‑RAG як тестовий набір.

## 4. Метрики й бенчмарки

### 4.1. Метрики

- **IR‑рівень:** Recall@k, nDCG@k, MRR — для оцінки retriever’а.
- **QA‑рівень:** Exact Match (EM), F1, іноді BLEU/ROUGE/BERTScore — для якості відповіді.
- **RAG‑специфічні:**
  - *Faithfulness* — чи відповіді узгоджені з retrieved‑контекстом;  
  - *Groundedness* — наскільки відповіді «заземлені» в джерелах;  
  - *Hallucination rate* — частка вигаданих фактів;  
  - *Citation coverage* — чи мають твердження посилання на джерела.
- Часто використовують **LLM‑as‑a‑judge** (інша модель оцінює якість відповіді).

### 4.2. Бенчмарки й датасети

- Класичні: **Natural Questions, TriviaQA, HotpotQA, KILT, BEIR**.
- Спеціалізовані для RAG: **FreshQA, RAGBench, T²‑RAGBench, CRAG, LegalBench‑RAG, WixQA, HaluBench** (для hallucinations).
- Фреймворки на кшталт **RAGAS** надають набір метрик і готові пайплайни оцінки.

## 5. Практичне застосування: фреймворки та сервіси

### 5.1. Open‑source

- **LangChain** — chains, агенти, retrievers, rerankers, memory; додаткові проєкти LangGraph (графи станів) і LangSmith (спостережуваність/оцінка).
- **LlamaIndex** — ingestion та індексація різнорідних джерел (файли, БД, API), деревоподібні/графові індекси, гнучкі query‑енджини.
- **Haystack** — production‑орієнтований фреймворк (pipelines, retrievers, generators, Kubernetes‑деплоймент) з інтеграціями в Elastic, OpenSearch, Weaviate, Qdrant.
- **DSPy, RAGFlow** — declarative / visual воркфлоу для проєктування й оптимізації RAG‑пайплайнів.

### 5.2. Хмарні сервіси

- **Azure AI Search / Azure AI Studio** — hybrid/vector search, інтеграція з Azure OpenAI; готові RAG‑шаблони.
- **AWS Bedrock + Kendra Knowledge Bases** — managed knowledge bases з конекторами до S3, SharePoint, Salesforce, Confluence; Bedrock‑LLM як генератор.
- **Google Vertex AI Search & Conversation** — hybrid search і чат‑/агентні сценарії з «grounded» відповідями й автоматичними цитатами джерел.

## 6. Обмеження та типові проблеми

- **Галюцинації й недостатня groundedness**: навіть із RAG LLM може вигадувати деталі або не посилатись на retrieved‑тексти.
- **Контекстні обмеження**: не все, що потрібно, поміщається в одне вікно контексту; необхідні компресія, multi‑hop, graph RAG.
- **Якість індексації й даних**: поганий чанкінг, шум, відсутність оновлення індексу сильно погіршують результати.
- **Оцінка в реальних умовах**: стандартні QA‑метрики не завжди відображають бізнес‑цінність; потрібні комбіновані метрики, людська валідація та доменно‑специфічні бенчмарки.

## 7. Самокритика та доопрацювання

Під час підготовки цього звіту я **свідомо перевіряв себе** на повноту й актуальність джерел у два етапи:

1. **Початкова версія** чернетки покривала базові речі (classic RAG, покращені retrievers, multi‑hop, agentic, memory/graph, типові метрики, кілька фреймворків і хмарних сервісів). Уже на цьому етапі джерела були сучасними (до 2024–2025 років), але:
   - Опис **Graph RAG** був надто загальним і майже без конкретних схем.  
   - **Agentic RAG** подавався як ідея, без чітких патернів (ReAct, plan‑and‑execute, self‑grading, multi‑agent).  
   - Блок про **оцінку й бенчмарки** торкався RAGAS і кількох датасетів, але не враховував новіші RAGBench‑подібні набори (T²‑RAGBench, HaluBench тощо).  
   - Перелік інструментів згадував DSPy/RAGFlow досить побіжно.

2. **Другий раунд (цільове доопрацювання)**:
   - Я окремо шукав сучасні роботи та огляди по **Graph RAG** (наприклад, *Medical Graph RAG: Towards Safe Medical LLM*, 2024) і деталізував типову схему: entity extraction → graph retrieval (neighbors/paths) → graph summarization → LLM.
   - Для **agentic RAG** я звернувся до статей на кшталт *Agentic RAG Explained* і survey *Engineering the RAG Stack* (2026), щоб явно назвати ключові патерни (ReAct, plan‑and‑execute, self‑grading, multi‑agent) і додати enterprise‑фокус (IT‑service desks, compliance, research).
   - Для **оцінки** я уточнив список сучасних бенчмарків і фреймворків: RAGBench, T²‑RAGBench, CRAG, LegalBench‑RAG, WixQA, HaluBench (Lynx), а також те, що в enterprise‑практиці комбінують synthetic тест‑сети (RAGAS/ARES), golden data й human review.
   - У блоці про **інструменти** я чіткіше розмежував OSS‑фреймворки (LangChain, LlamaIndex, Haystack, DSPy, RAGFlow) і managed‑сервіси хмар (Azure, AWS, GCP), підкресливши, що останні надають не лише векторне сховище, а й оркестрацію, безпеку, спостережуваність.

У результаті:
- Поточний звіт **короткий, але враховує сучасні (станом до 2025–початку 2026) уявлення про RAG**, включно з agentic і graph‑орієнтованими варіантами, новітні метрики та бенчмарки.
- Якщо в майбутньому з’являться якісно нові архітектури чи стандартизовані бенчмарки, цей звіт варто буде оновити, але станом на зараз він відображає актуальний стан практики, наскільки це дозволяють доступні джерела.
